# 📑 S14: SOA via Interfaces (Code-First gRPC)

**Status:** 📝 Draft  
**Owner:** Engineering Team  
**Reviewers:** Community / Architects

---

## 1. Goal
Provide a transparent, "code-first" Service-Oriented Architecture (SOA) layer for the Dext Framework. The goal is to allow developers to expose explicit Delphi business logic interfaces automatically as high-performance endpoints (primarily over gRPC/Protobuf, with REST/JSON fallback) with zero boilerplate, eliminating the need to write manual controllers, HTTP routing, or manual `.proto` definitions.

## 2. Context & Rationale
Legacy Delphi applications often contain massive, monolithic business rules hidden behind simple interfaces. To transition to a modern architecture (Web/Mobile/Microservices):
* Building REST conventions (Controllers/Minimal APIs) can represent high friction for migrating legacy systems.
* An RPC (Remote Procedure Call) approach based on native Interfaces (e.g., `ICalculadoraService`) is natural to Delphi developers and provides a zero-friction modernization path.
* By using **gRPC** and **Protobuf** as the core transport — combined with Dext's High-Performance Reflection and Zero-Allocation spans — Dext will deliver unparalleled performance natively over HTTP/2. This mirrors the DX of `protobuf-net.Grpc` in the .NET ecosystem.

## 3. Core Components

### 3.1. Code-First Service Mapping
* Developers define an interface decorated with attributes (e.g., `[ServiceContract]`) or inheriting from a marker interface.
* Server registration requires a single line: `app.MapGrpcService<ICalculadora, TCalculadoraImpl>();`.
* No `.proto` schema files need to be authored; the framework infers the schema directly from the Delphi interface using Dext's RTTI caching.

### 3.2. High-Performance Deserialization Pipeline
* Request parameters are hydrated directly from the Protobuf binary buffer (or JSON) into the method arguments, avoiding intermediate string allocations.
* Deep integration with Dext.DI: classes implementing the interfaces are managed through the container (Transient, Scoped, Singleton).
* Built on top of the robust meta-engine designed in [S07: High-Perf-Reflection](S07-High-Perf-Reflection.md).

## 4. Technical Requirements

### 4.1. Contract Abstraction
```pascal
[ServiceContract]
ICalculadoraService = interface(IInvokable)
  ['{GUID}']
  function Somar(A, B: Integer): Integer;
end;
```
* Methods defined in the interface are automatically mapped as endpoints (e.g. `/CalculadoraService/Somar`).
* Support for asynchronous execution using interfaces that return Task-like structures or callbacks.

### 4.2. Transport Capabilities
* Architecturally bonded with [S02: Modernizer (gRPC & Protobuf)](S02-Modernizer-gRPC.md).
* Provides dynamic `schema generation`, being able to export a virtual `.proto` file to allow external consumers (e.g., Flutter, Vue, React Native) to auto-generate client stubs.
* Unified pipeline: SOA interfaces operate alongside standard Dext Minimal APIs and MVC endpoints seamlessly within the same web application pipeline.

## 5. Acceptance Criteria
- [ ] A developer can define, register, and expose an Interface as a service simply by invoking the application map method.
- [ ] Request parameters are deserialized natively from Protobuf leveraging zero-allocation parsing mechanisms.
- [ ] An external non-Delphi client can successfully invoke the service and receive a strongly-typed response.
- [ ] Documentation and example project (`S14-SOA-Demo`) provided to demonstrate legacy transformation.
