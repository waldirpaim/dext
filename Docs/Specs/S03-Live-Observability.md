# 📑 S03: Live Observability Dashboard Specification

**Status:** ✅ Functional Core (Bridge Implemented)  
**Owner:** Engineering Team  
**Reviewers:** Community / Architects

---

## 1. Goal
Provide a real-time debugging and monitoring interface (Dashboard) to visualize the internal behavior of a Dext application. Focused on local development and high-level tracing.

## 2. Main Features (dotpas Observability)

### 2.1. Request & SQL Telemetry
- **DiagnosticSource**: A low-latency event system that decoupled components (Web Server, ORM) use to publish internal events.
- **SQL Trapping**: Capture SQL commands sent by the ORM. The actual SQL instruction is extracted and formatted for readability.
- **Logging Bridge**: Integration with `ILogger` ensures that telemetry events are visible in the console/logs.
- **Visual Timeline**: See the order of execution (Middleware -> Handler -> ORM -> Response).

### 2.2. Exception & Stack Trace
- Automatic capture of unhandled exceptions via **Telemetry Channels**.
- **Deep Stack Trace**: Integration with `FastMM5` or JCL to show the exact source line of the error.

### 2.3. Instrumentation via Dynamic Proxy (Diagnostic Interceptor)
- Use dynamic proxies (Interceptors) to trace business methods (Interfaces/Classes) without modifying the original code.
- **Constraint**: This feature should only be active in **DEBUG** mode to avoid production performance penalties.

### 2.4. DataAPI Transparency
- **Virtual Implementation Tracing**: Since DataAPIs are generated at runtime, the telemetry must capture the internal stages:
    - Query Parameter Parsing (Filters, Pagination).
    - Mapping of URL field names to Entity Properties.
    - Security checks (Authentication/Roles) performed by the handler.

---

## 3. Technical Requirements

### 3.1. User Interface
- A built-in web dashboard (Dark Theme) accessible at `/_dext/dashboard` or similar.
- Use of **WebSockets** (Dext.Web.Hubs) for real-time log streaming.

### 3.2. Storage
- Volatile in-memory storage (Capped at N events) for local dev.
- Optional adapter for persistent logging (ElasticSearch/OpenTelemetry).

## 4. Acceptance Criteria
- [x] Developer can see the SQL being generated by an endpoint in the console/logs.
- [ ] Developer can open the dashboard and see the events in real-time (Next Stage).
- [ ] Stack trace correctly points to the failing unit/method.
- [ ] DataAPI internal steps (Filter parsing, mapping) are visible in the telemetry stream.
- [x] Minimal overhead when the observers are closed.

---
 
## 5. Implementation Progress
- **Infrastructure**: Core `TDiagnosticSource` (observer pattern) implementation in `Dext.Logging.Telemetry.pas`.
- **Instrumentation**:
    - **Database**: `TDbSet<T>` captures SQL queries, inserts, updates, and deletes with affected row counts. Actual SQL extracted from payload.
    - **Web**: `TDextPipeline` captures incoming HTTP requests and latency.
    - **Bridge**: `TLoggingTelemetryObserver` routes events to `ILogger`.
- **Verification**: Verified via `Web.TicketSales` example.
- **Next Steps**: Implementation of the built-in web dashboard (`/_dext/dashboard`) using Vue.js/Next.js and WebSockets.
