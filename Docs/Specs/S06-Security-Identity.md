# 📑 S06: Security & Identity (OAuth2/OIDC) Specification

**Status:** 📝 Draft  
**Owner:** Engineering Team  
**Reviewers:** Community / Architects

---

## 1. Goal
Establish a top-tier security layer for Dext applications. Provide native support for modern authentication and authorization standards (OAuth2, OpenID Connect, JWT) to make Dext enterprise-ready.

## 2. Core Features

### 2.1. JWT Strategy
- Built-in support for generating and validating **JSON Web Tokens (JWT)**.
- Support for multiple signing algorithms (HS256, RS256).
- **Automatic Refresh Token** management.

### 2.2. Dext Identity Server (Internal)
- A specialized middleware/module to act as an Identity Provider.
- Support for `Authorization Code`, `Client Credentials`, and `Implicit` flows.
- User management abstractions (IdentityUser, IdentityRole).

### 2.3. External Identity Providers
- Plug-and-play middleware for:
    - **Google**, **Microsoft Entra ID (Azure)**, **GitHub**.
    - OIDC compatible servers (Keycloak, Auth0).

### 2.4. Policy-Based Authorization
- Move beyond simple Roles to **Policies** (e.g., `[Authorize(Policy = "MinimumAge21")]`).

---

## 3. Technical Requirements

### 3.1. Crypto Engine
- Use **Dext.Core.Crypto** (OpenSSL 3.0 or native OS APIs) for high-performance signature verification.
- Zero-allocation JWT parsing.

### 3.2. Middleware Integration
- Strict integration with the `Dext.Web` pipeline.
- Automatic injection of `ClaimsPrincipal` into the request context.

## 4. Acceptance Criteria
- [ ] A developer can secure an API with 3 lines of code: `.AddAuthentication.AddJwtBearer(...)`.
- [ ] Support for "Social Login" (Login with Google) out of the box.
- [ ] Compatible with standard JWT tools and validators.
